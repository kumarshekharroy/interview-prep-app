# Revision System

This file defines how to review and retain material during the 24-week plan and after applications begin.

## Revision Principles

1. Revise weak areas more often than strong areas.
2. Speak interview answers aloud.
3. Retake mistakes under time pressure.
4. Use spaced repetition for concepts.
5. Use active recall, not passive rereading.

## Daily Revision Routine

Use at the end of every study day.

| Time | Action |
| ---: | --- |
| 5 minutes | write what was learned |
| 5 minutes | write one mistake |
| 5 minutes | answer the daily interview prompt aloud |
| 5 minutes | update weak-area log if needed |

## Weekly Revision Routine

Use every Sunday.

1. Review all daily artifacts.
2. Complete weekly assessment.
3. Update weekly score tracker.
4. Add weak areas.
5. Schedule recovery tasks.
6. Retake the weakest DSA problem.
7. Speak one system design answer.
8. Speak one behavioral answer.

## Spaced Revision Schedule

| When | What To Review |
| --- | --- |
| same day | daily concept, task, mini-test |
| next day | mistakes and weak concepts |
| 3 days later | key interview prompts |
| 7 days later | weekly review and assessment |
| 30 days later | monthly checkpoint and project defense |

## DSA Revision Method

For every DSA problem:

1. Name pattern.
2. State brute force.
3. State optimized approach.
4. Code.
5. Test edge cases.
6. State complexity.
7. Write one bug to avoid.

### DSA Retake Ladder

| Failure Type | Retake Action |
| --- | --- |
| pattern not recognized | solve two easier same-pattern problems |
| edge case missed | write five edge cases before recoding |
| implementation bug | recode from scratch next day |
| explanation weak | redo aloud without code |
| time exceeded | redo with stricter planning cap |

## System Design Revision Method

For every system design prompt:

1. Requirements.
2. Scope.
3. Scale.
4. APIs.
5. Data.
6. Architecture.
7. Critical flows.
8. Reliability.
9. Security.
10. Observability.
11. Deployment.
12. Tradeoffs.

### System Design Retake Ladder

| Miss | Retake Action |
| --- | --- |
| skipped requirements | redo first 5 minutes only |
| weak data model | redraw schema and indexes |
| weak service choice | redo service selection table |
| no observability | add dashboard and alerts |
| no tradeoffs | compare two alternatives |
| rambling | redo with 10-minute time box |

## Backend/Azure Revision Method

Use rapid 90-second answers for:

* ASP.NET Core request pipeline.
* EF Core query performance.
* SQL indexing.
* outbox pattern.
* idempotent consumer.
* Service Bus queue vs topic.
* RabbitMQ exchanges.
* Event Grid vs Event Hubs.
* App Service vs Container Apps.
* Key Vault and managed identity.
* Application Insights vs OpenTelemetry.

Each answer must include:

* definition.
* example.
* failure mode.
* tradeoff.

## Behavioral Revision Method

For every behavioral story:

1. Situation in 20 seconds.
2. Task in 10 seconds.
3. Action in 60-90 seconds.
4. Result in 20 seconds.
5. Reflection in 10 seconds.

If the story exceeds 3 minutes, shorten setup.

## Career Revision Method

Review weekly:

* resume top bullets.
* LinkedIn headline and About.
* recruiter templates.
* referral targets.
* salary scripts.
* application tracker.
